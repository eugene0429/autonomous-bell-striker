# YOLO26n Bell Detection — Training Pipeline Design

**Date:** 2026-04-27
**Owner:** eugene (sim2real)
**Status:** Approved (brainstorming) → ready for plan

---

## 1. Goal

Train a YOLO26n single-class detector for the vibrating bell target used in
Capstone 2026 Phase 2 aiming. Output is a deployable `best.pt` that runs on
Pi 5 (CPU) at acceptable real-time rate for the aiming loop.

Non-goals:
- Multi-class detection
- ONNX/TFLite export (separate task once weights are validated)
- Data augmentation tuning beyond ultralytics defaults

---

## 2. Dataset Inventory

| Item | Value |
|---|---|
| Root | `perception/dataset/` |
| Image dirs | `imgs/scenario_{01..06}_<dist>m_<pos>/` (640×480 JPG) |
| Label dirs | `labels/{01..06}_labels/` (YOLO normalized txt) |
| Scenarios | 2m·4m × left/middle/right (6 total) |
| Images per scenario | 150 |
| Total images | 900 |
| Total labels | 889 (11 unlabeled images → skip) |
| Classes | 1 (id=0 = `bell`) |

**Naming asymmetry:** image dir `scenario_NN_*` ↔ label dir `NN_labels`. The
NN prefix is the join key.

---

## 3. Split Strategy — Stratified Random (Approach A)

Within each scenario directory, deterministically shuffle the labeled images
and split **80 / 10 / 10** for train / val / test. Net counts:

- ~711 train / ~88 val / ~89 test (varies ±1 per scenario from 11 unlabeled)

All 6 scenarios appear in every split, so val/test measure performance across
the full operating envelope (2~4 m, L/M/R) rather than generalization to
unseen scenarios. Justified because the deployment scenario set IS this grid.

Seed: 42, fixed for reproducibility.

---

## 4. Architecture

Two-script layout under `perception/training/`:

```
perception/training/
├── prepare_dataset.py      # one-time data prep
├── train.py                # entrypoint: train + test-eval
├── dataset.yaml            # generated by prepare_dataset.py
├── data/                   # generated symlink tree (gitignored)
│   ├── images/{train,val,test}/*.jpg
│   └── labels/{train,val,test}/*.txt
└── runs/                   # ultralytics output (gitignored)
```

Splitting prep from training lets the user re-run training with different
hyperparameters without re-shuffling the dataset.

### 4.1 `prepare_dataset.py`

Responsibilities:
1. Discover the 6 image dirs and resolve each to its label dir via the NN key
2. For each scenario, enumerate images and pair with `<stem>.txt` in the
   matching label dir; drop pairs where the txt is missing or zero-byte
3. Stratified 80/10/10 split per scenario (seeded)
4. Build symlink tree under `data/{images,labels}/{train,val,test}/`
   - Filenames are unique across scenarios (timestamp-based) so flattening
     is safe; verify uniqueness and abort if collision
5. Write `dataset.yaml` consumed by ultralytics

CLI:
- `--rebuild` — wipe `data/` and `dataset.yaml` and regenerate
- Default: idempotent (skip if `dataset.yaml` exists)

`dataset.yaml` shape:
```yaml
path: <abs path to perception/training/data>
train: images/train
val:   images/val
test:  images/test
names:
  0: bell
```

### 4.2 `train.py`

Responsibilities:
1. Ensure dataset is prepared (call into prepare logic if `dataset.yaml`
   missing — no `--rebuild` here, just lazy build)
2. Load `yolo26n.pt` pretrained weights
3. Train with the hyperparameters in §5
4. After training, run `model.val(split='test')` on the held-out test set
   using `best.pt`, print mAP@0.5 / mAP@0.5:0.95 / per-image latency

CLI overrides (all optional):
- `--epochs` (default 150)
- `--imgsz` (default 640)
- `--batch` (default -1 = auto)
- `--device` (default 0)
- `--name` (default `bell_yolo26n`)
- `--workers` (default 8)
- `--no-test` — skip post-training test evaluation

Outputs land in `perception/training/runs/detect/<name>/`.

## 5. Training Hyperparameters

| Param | Value | Reason |
|---|---|---|
| model | `yolo26n.pt` | Pi 5 CPU deploy; 2.6M params, 5.3 MB |
| imgsz | 640 | Matches native 640×480 (auto-pad to 640×640) |
| epochs | 150 | small dataset, early-stop will cut |
| patience | 30 | early stop if val mAP plateaus 30 epochs |
| batch | -1 (auto) | 3090 24 GB headroom |
| device | 0 | single GPU |
| optimizer | auto | ultralytics picks SGD/AdamW |
| amp | true | fp16 default |
| cos_lr | true | smoother LR decay for short schedules |
| close_mosaic | 10 | mosaic off in last 10 epochs to settle bbox |
| single_cls | true | force single class even though labels are id=0 |
| seed | 42 | reproducibility |

Augmentation: ultralytics defaults (mosaic, hsv, fliplr=0.5). The dataset
already covers distance & lateral position diversity, so no custom aug.

---

## 6. Model Size Justification

**26n is the right choice** given:
- Pi 5 CPU inference budget — 26s is ~3× slower, 26m unfeasible at real time
- Single class + ~711 train images + simple bell geometry → 26n capacity
  sufficient; larger model risks overfit
- 5.3 MB checkpoint trivializes deployment

Escalation rule: if test mAP@0.5 < 0.95 OR field deployment shows missed
detections, retrain with `yolo26s.pt` (same script, swap one arg). Don't
preemptively go bigger.

---

## 7. Validation & Acceptance

- `prepare_dataset.py` runs idempotently; `--rebuild` produces identical
  splits given the same seed
- `train.py` completes a full epoch on the prepared data without error
- Final `best.pt` achieves mAP@0.5 ≥ 0.90 on the test split (soft target —
  if not met, treat as a flag to investigate dataset, not a hard fail)
- Test-eval section prints per-image latency on the training GPU (sanity
  check; Pi 5 latency measured separately during deployment)

---

## 8. Out of Scope

- Pi 5 latency benchmark (lives in deployment work)
- ONNX / TFLite / NCNN export
- TensorBoard or W&B integration (ultralytics built-in CSV/png plots
  sufficient for this scope)
- Multi-GPU training (DDP) — single 3090 is plenty

---

## 9. Risks

| Risk | Mitigation |
|---|---|
| Filename collision when flattening 6 dirs | Verify uniqueness in prep; abort with clear error if collision |
| YOLO26 weight download fails (no internet on training box) | Confirm pretrained weight already on disk before run; the prior probe downloaded `yolo26n.pt` to repo root — move to a stable cache path |
| Test mAP misleadingly high (same scenarios in train and test) | Documented as known limitation; field tests are the real generalization signal |
| `single_cls=True` masks label corruption (multi-class typo) | Prep step already validates label format; print warning if any label has class != 0 |
